Билет №1. Актуальность автоматизации бизнес-процессов.
Краткий ответ
Автоматизация бизнес-процессов — это перевод повторяющихся операций организации в управляемую цифровую форму: через информационные системы, скрипты, интеграции, RPA, CRM, ERP, BPM-системы и другие средства. Ее актуальность связана с тем, что бизнесу нужно быстрее обрабатывать данные, снижать количество ручных ошибок и получать прозрачную картину работы подразделений.
Основные положения
•	ускоряет выполнение типовых операций: оформление заявок, учет заказов, расчет показателей, подготовку отчетности;
•	снижает зависимость результата от человеческого фактора, потому что правила выполнения процесса фиксируются в системе;
•	дает руководству измеримые показатели: сроки выполнения задач, узкие места, загрузку сотрудников, количество ошибок;
•	облегчает масштабирование: при росте количества операций не всегда требуется пропорционально увеличивать штат;
•	помогает интегрировать сайт, склад, бухгалтерию, CRM, аналитику и другие системы в единый процесс.
Пояснение
Автоматизация не должна просто переносить хаос в компьютер. Перед внедрением описывают процесс, убирают лишние этапы, определяют ответственных, входные и выходные данные, а затем выбирают инструмент. Хорошая автоматизация повышает качество, скорость и управляемость процесса; плохая закрепляет ошибки и делает их массовыми.
Вывод
Автоматизация актуальна потому, что превращает бизнес-процесс из набора ручных действий в контролируемую, повторяемую и анализируемую систему.
 
Билет №2. Инструментальные средства разработки программного обеспечения.
Краткий ответ
Инструментальные средства разработки программного обеспечения — это программы и сервисы, которые помогают разработчику создавать, проверять, собирать, запускать, сопровождать и документировать программный продукт. Они охватывают весь жизненный цикл ПО: от написания кода до тестирования, развертывания и мониторинга.
Основные положения
•	редакторы кода и ИСР: Visual Studio, IntelliJ IDEA, PyCharm, VS Code, Eclipse;
•	компиляторы и интерпретаторы, которые переводят исходный код в исполняемую форму или выполняют его;
•	системы сборки и пакетные менеджеры: Maven, Gradle, npm, pip, NuGet;
•	системы контроля версий: Git, SVN;
•	средства тестирования: JUnit, NUnit, pytest, Selenium, Postman;
•	инструменты анализа качества: линтеры, статические анализаторы, профилировщики, средства покрытия тестами;
•	CI/CD-сервисы: GitHub Actions, GitLab CI, Jenkins.
Пояснение
Выбор инструментов зависит от языка программирования, архитектуры проекта, команды и требований к продукту. Например, для Java часто используют IntelliJ IDEA, Maven или Gradle и JUnit; для веб-разработки — VS Code, Node.js, npm, ESLint и браузерные DevTools.
Вывод
Инструментальные средства повышают производительность разработчика, уменьшают число ошибок и делают процесс разработки воспроизводимым.
 
Билет №3. Интегрированная среда разработки программного обеспечения. Назначение и основные возможности ИСР.
Краткий ответ
Интегрированная среда разработки, или ИСР, — это комплексное программное средство, объединяющее основные функции для разработки ПО в одном интерфейсе. Обычно ИСР включает редактор кода, компилятор или интерпретатор, средства запуска, отладчик, инструменты работы с проектом и интеграцию с системами контроля версий.
Основные положения
•	редактирование кода: подсветка синтаксиса, автодополнение, шаблоны, форматирование;
•	навигация по проекту: переход к определению класса, метода или переменной, поиск использований;
•	сборка и запуск проекта, настройка конфигураций запуска, подключение библиотек;
•	отладка: точки останова, пошаговое выполнение, просмотр переменных, стек вызовов;
•	работа с Git: commit, diff, merge, pull, push, просмотр истории изменений;
•	запуск тестов и просмотр результатов;
•	анализ качества: инспекции кода, быстрые исправления, рефакторинг.
Пояснение
Главное назначение ИСР — сократить время между написанием кода, проверкой и исправлением ошибки. Вместо переключения между множеством отдельных программ разработчик получает единое рабочее пространство. При этом ИСР не заменяет знания языка и архитектуры: она помогает быстрее применять эти знания на практике.
Вывод
ИСР является центральным рабочим инструментом разработчика, потому что объединяет создание, запуск, проверку и сопровождение кода.
 
Билет №4. Понятие интеграции программного обеспечения. Цели и назначение интеграции.
Краткий ответ
Интеграция программного обеспечения — это объединение отдельных программ, модулей, сервисов или информационных систем так, чтобы они могли обмениваться данными и совместно выполнять бизнес-задачи. Интеграция нужна, когда одна система не может самостоятельно покрыть весь процесс или когда данные должны проходить между разными участниками процесса.
Основные положения
•	обеспечить согласованный обмен данными между системами;
•	уменьшить ручной ввод и количество ошибок при копировании данных;
•	ускорить процессы за счет автоматической передачи информации;
•	создать единую информационную среду, где разные системы выполняют свои функции;
•	использовать API, обмен файлами, очереди сообщений, webhooks, общую базу данных, ETL-процессы или сервисную шину.
Пояснение
При интеграции важно определить формат данных, протокол обмена, правила обработки ошибок, безопасность, права доступа, частоту обмена и ответственность систем. Если эти вопросы не решить заранее, интеграция может стать нестабильной: данные будут теряться, дублироваться или приходить в неверном формате.
Вывод
Интеграция превращает набор отдельных программ в единую систему, способную поддерживать сквозной бизнес-процесс.
 
Билет №5. Транспортные протоколы. Сериализация и десериализация данных. Основные форматы представления данных для обмена между информационными системами.
Краткий ответ
Транспортные протоколы и форматы данных определяют, как информационные системы передают сообщения друг другу и как эти сообщения представлены внутри передачи. В интеграции важно разделять канал обмена и структуру данных: протокол отвечает за доставку, а формат — за представление информации.
Основные положения
•	TCP обеспечивает надежную доставку с контролем порядка пакетов, UDP быстрее, но не гарантирует доставку и порядок;
•	HTTP/HTTPS часто используется для веб-API, WebSocket — для постоянного двустороннего соединения, AMQP и MQTT — для обмена сообщениями;
•	сериализация — преобразование объекта или структуры данных в формат, пригодный для передачи или хранения;
•	десериализация — обратное преобразование полученных данных в объект программы;
•	JSON легкий и удобный для веб-API, XML строгий и расширяемый, YAML удобен для конфигураций, CSV подходит для таблиц;
•	Protocol Buffers, Avro и MessagePack — бинарные форматы, которые обычно компактнее и быстрее текстовых.
Пояснение
При выборе протокола и формата учитывают надежность, скорость, читаемость, поддержку языками программирования, безопасность, совместимость версий и объем данных. Например, публичный REST API часто использует HTTPS и JSON, а высоконагруженный внутренний сервис может использовать gRPC и Protocol Buffers.
Вывод
Протокол отвечает за передачу сообщения, а сериализация — за превращение данных в понятный для обмена формат.
 
Билет №6. Основные понятия системы контроля версий. Понятие репозитория проекта, структура проекта.
Краткий ответ
Система контроля версий — это инструмент, который хранит историю изменений файлов проекта и позволяет возвращаться к предыдущим состояниям, сравнивать изменения, объединять работу разных разработчиков и выпускать версии продукта. Самый распространенный пример — Git.
Основные положения
•	репозиторий — хранилище проекта с историей изменений;
•	коммит — зафиксированное состояние изменений с автором, датой и сообщением;
•	ветка — отдельная линия разработки для функции, исправления или эксперимента;
•	слияние — объединение изменений из разных веток;
•	конфликт возникает, когда разные изменения затрагивают одни и те же участки кода;
•	удаленный репозиторий на сервере позволяет команде обмениваться изменениями;
•	структура проекта обычно включает исходный код, тесты, документацию, конфигурации, файлы сборки и служебные каталоги.
Пояснение
Хорошая структура репозитория делает проект понятным: по названиям каталогов видно, где находится код приложения, где тесты, где инструкции по запуску и где настройки сборки. Файлы, которые генерируются автоматически или содержат локальные настройки разработчика, обычно исключают через .gitignore.
Вывод
Система контроля версий сохраняет историю проекта и делает командную разработку управляемой.
 
Билет №7. Подходы к управлению версиями программного обеспечения.
Краткий ответ
Управление версиями программного обеспечения — это правила, по которым команда фиксирует изменения, выпускает релизы, поддерживает старые версии и сообщает пользователям о совместимости. Оно связано не только с Git, но и с организацией жизненного цикла продукта.
Основные положения
•	централизованный подход: основной репозиторий находится на сервере, пример — SVN;
•	распределенный подход: каждый разработчик имеет полноценную копию репозитория, пример — Git;
•	ветвление по функциям: под каждую задачу создается отдельная feature-ветка;
•	Git Flow выделяет main, develop, feature, release и hotfix-ветки;
•	trunk-based development предполагает частую интеграцию небольших изменений в основную ветку;
•	семантическое версионирование использует формат MAJOR.MINOR.PATCH;
•	теги релизов, changelog и hotfix-процессы помогают управлять выпуском продукта.
Пояснение
Чем больше пользователей и интеграций у продукта, тем важнее предсказуемость версий. Если разработчики ломают совместимость без предупреждения, зависимые системы и пользователи получают сбои. Поэтому управление версиями связано с документацией, тестированием и политикой релизов.
Вывод
Управление версиями помогает команде развивать продукт без потери контроля над изменениями и совместимостью.
 
Билет №8. Совместная разработка ПО. Преимущество использования совместной разработки проектов.
Краткий ответ
Совместная разработка ПО — это организация работы, при которой несколько специалистов одновременно создают, проверяют и сопровождают один программный продукт. В ней участвуют разработчики, тестировщики, аналитики, дизайнеры, DevOps-инженеры и заказчики.
Основные положения
•	техническая основа совместной разработки — общий репозиторий и система контроля версий;
•	команда использует задачи, ветки, pull request, code review, автоматические тесты и CI/CD;
•	большие задачи выполняются быстрее, потому что разные участники работают параллельно;
•	качество повышается за счет взаимной проверки кода и автоматических проверок;
•	знания распределяются внутри команды, и продукт меньше зависит от одного разработчика;
•	история коммитов, обсуждений и задач помогает понять причины принятых решений.
Пояснение
Для эффективной совместной разработки нужны договоренности: стиль кода, правила ветвления, формат коммитов, обязательность тестов, порядок ревью, ответственность за модули. Без этих правил параллельная работа может привести к конфликтам, дублированию решений и неустойчивой архитектуре.
Вывод
Совместная разработка позволяет создавать более сложные продукты быстрее и надежнее, если процесс работы хорошо организован.
 
Билет №9. Модульный принцип программирования. Особенности, преимущества и недостатки модульной разработки ПО.
Краткий ответ
Модульный принцип программирования — это подход, при котором программа разбивается на относительно самостоятельные части: модули, классы, пакеты, библиотеки или сервисы. Каждый модуль отвечает за свою область функциональности и взаимодействует с другими через четко определенный интерфейс.
Основные положения
•	модульность основана на разделении ответственности;
•	модули можно разрабатывать и тестировать независимо;
•	хороший модуль можно повторно использовать в других частях проекта;
•	ошибку проще искать в ограниченной части системы;
•	модульность повышает тестируемость, потому что модуль можно проверять через unit-тесты;
•	недостаток — необходимость проектировать понятные интерфейсы;
•	слишком большое дробление может усложнить интеграцию и навигацию по проекту.
Пояснение
Модульность работает лучше, когда соблюдаются принципы высокой связности внутри модуля и слабой связанности между модулями. Это означает, что элементы одного модуля должны быть логически близки, а зависимости от других модулей — минимальны и понятны.
Вывод
Модульная разработка делает систему более понятной и управляемой, но требует дисциплины в проектировании интерфейсов.
 
Билет №10. Возможные проблемы интеграции модулей в программный продукт.
Краткий ответ
Интеграция модулей — это этап, на котором отдельно разработанные части программного продукта соединяются в единую систему. Даже если каждый модуль работает сам по себе, при соединении могут возникнуть ошибки из-за различий в интерфейсах, данных, зависимостях и ожиданиях разработчиков.
Основные положения
•	несовместимость интерфейсов: один модуль передает параметры в одном формате, другой ожидает другой;
•	несогласованные версии: модуль обновлен, а зависимые части используют старый контракт;
•	разные правила обработки ошибок: код ошибки, исключение или молчаливое игнорирование;
•	конфликт версий библиотек и зависимостей;
•	проблемы производительности при массовых вызовах;
•	гонки данных, блокировки и проблемы конкурентности;
•	недостаточная документация и разное понимание допустимых значений.
Пояснение
Для снижения рисков используют контрактное проектирование, API-документацию, интеграционные тесты, тестовые стенды, mock-сервисы, версионирование интерфейсов, CI-сборки и постепенную интеграцию. Чем раньше модули начинают проверяться вместе, тем дешевле исправлять несовместимости.
Вывод
Основная сложность интеграции модулей состоит в согласовании поведения, данных и ожиданий между частями системы.
 
Билет №11. Подходы к отладке программных продуктов.
Краткий ответ
Отладка программного продукта — это процесс поиска, анализа и устранения причин неправильного поведения программы. Ее задача — понять не только где произошла ошибка, но и почему программа пришла к такому состоянию.
Основные положения
•	пошаговая отладка позволяет выполнять программу по строкам или функциям;
•	точки останова останавливают выполнение в выбранных местах;
•	логирование фиксирует важные события и ошибки в журнале;
•	метод гипотез помогает проверять возможные причины минимальными экспериментами;
•	бинарный поиск по коду или данным сокращает область поиска ошибки;
•	тесты воспроизводят дефект и защищают от его повторного появления;
•	профилирование применяется для проблем скорости, памяти и ресурсов.
Пояснение
Хорошая отладка начинается с воспроизведения ошибки. Нужно знать входные данные, окружение, версию программы, ожидаемый и фактический результат. Исправление наугад опасно: оно может скрыть симптом, но оставить настоящую причину.
Вывод
Отладка — это исследование поведения программы с помощью инструментов, логики и контролируемых проверок.
 
Билет №12. Точки останова. Назначение и использование. Отладочные классы. Назначение и использование.
Краткий ответ
Точка останова — это специальная отметка в коде, при достижении которой программа приостанавливает выполнение в режиме отладки. Она позволяет изучить состояние приложения в конкретный момент: значения переменных, стек вызовов, текущий объект, состояние потоков и другие данные.
Основные положения
•	обычная строковая точка останова срабатывает при выполнении выбранной строки;
•	условная точка срабатывает только при выполнении заданного условия;
•	точка останова по исключению останавливает программу при возникновении исключения;
•	точка трассировки может выводить диагностическое сообщение без остановки программы;
•	точки останова помогают проверить, попадает ли программа в нужную ветку кода;
•	отладочные классы и API: Debug, Trace, Logger, Assert, Stopwatch;
•	они применяются для вывода сообщений, проверки предположений, измерения времени и диагностики.
Пояснение
Отладочные классы полезны, когда интерактивный отладчик недоступен: например, на сервере, в тестовом окружении или у пользователя. Assert помогает обнаружить нарушение логического условия во время разработки, а logger сохраняет информацию об ошибках и важных событиях при работе приложения.
Вывод
Точки останова нужны для интерактивного анализа программы, а отладочные классы — для встроенной диагностики и контроля выполнения.
 
Билет №13. Подходы к тестированию программных продуктов.
Краткий ответ
Тестирование программного продукта — это проверка соответствия программы требованиям и ожиданиям пользователей. Оно помогает обнаружить дефекты, снизить риски и подтвердить, что система выполняет нужные функции в заданных условиях.
Основные положения
•	по уровню выделяют модульное, интеграционное, системное и приемочное тестирование;
•	по способу выполнения тестирование бывает ручным и автоматизированным;
•	по знанию внутреннего устройства системы выделяют черный, белый и серый ящик;
•	функциональное тестирование проверяет заявленные функции;
•	нефункциональное проверяет производительность, безопасность, удобство, надежность и совместимость;
•	регрессионное тестирование проверяет, не сломали ли новые изменения старую функциональность;
•	дымовое тестирование быстро подтверждает, что сборка пригодна для дальнейшей проверки.
Пояснение
Тестирование не доказывает абсолютное отсутствие ошибок, но снижает вероятность критических дефектов. На практике комбинируют разные подходы: быстрые unit-тесты запускают часто, интеграционные тесты проверяют связи, ручное тестирование помогает оценить сценарии пользователя и удобство интерфейса.
Вывод
Тестирование — это систематическая проверка качества ПО на разных уровнях и с разных точек зрения.
 
Билет №14. Модульное тестирование. Использование тестовых классов.
Краткий ответ
Модульное тестирование, или unit-тестирование, — это проверка отдельной минимальной части программы: функции, метода, класса или модуля. Цель — убедиться, что эта часть корректно работает изолированно от остальной системы.
Основные положения
•	unit-тесты обычно быстрые и запускаются часто;
•	тест проверяет конкретное ожидаемое поведение: входные данные, действие и результат;
•	тестовые классы группируют тесты по проверяемому классу или модулю;
•	assert-проверки сравнивают ожидаемый и фактический результат;
•	mocks, stubs и fakes заменяют базу данных, сеть, файловую систему или внешние сервисы;
•	хороший unit-тест независим, повторяем, понятен и не зависит от порядка запуска.
Пояснение
Классическая структура теста: Arrange, Act, Assert. На этапе Arrange подготавливают данные и объект, на этапе Act выполняют действие, на этапе Assert проверяют результат. Такая структура делает тест читаемым и облегчает поиск причины сбоя.
Вывод
Модульное тестирование помогает быстро обнаруживать ошибки в отдельных частях программы и безопаснее изменять код.
 
Билет №15. Актуальность использования ручного тестирования программных продуктов.
Краткий ответ
Ручное тестирование — это проверка программы человеком без полного автоматического выполнения сценария. Несмотря на развитие автотестов, ручное тестирование остается актуальным, потому что не все свойства продукта легко формализовать в коде.
Основные положения
•	полезно на ранних этапах, когда требования и интерфейс часто меняются;
•	человек лучше замечает проблемы удобства, визуальные дефекты и непонятные тексты;
•	исследовательское тестирование помогает находить ошибки вне заранее описанных сценариев;
•	ручная проверка нужна для приемки новых функций заказчиком или аналитиком;
•	некоторые сценарии сложно автоматизировать: оборудование, визуальное восприятие, нестандартные состояния;
•	ручное тестирование помогает уточнять требования и находить противоречия.
Пояснение
Недостатки ручного тестирования — низкая скорость повторения, зависимость от внимательности человека и сложность проверки больших регрессионных наборов. Поэтому повторяемые стабильные сценарии автоматизируют, а новые и неоднозначные исследуют вручную.
Вывод
Ручное тестирование актуально там, где важны человеческая оценка, гибкость и исследование поведения продукта.
 
Билет №16. Интеграционное тестирование. Особенности и актуальность применения.
Краткий ответ
Интеграционное тестирование — это проверка взаимодействия нескольких модулей, компонентов или систем. Если модульное тестирование отвечает на вопрос «правильно ли работает часть?», то интеграционное — «правильно ли части работают вместе?».
Основные положения
•	проверяются интерфейсы между модулями: данные, форматы, ошибки, последовательность вызовов;
•	проверяется работа с базой данных, API, очередями сообщений, файловой системой и внешними сервисами;
•	выявляются проблемы, не видимые в unit-тестах: несовместимые контракты, неверные настройки, ошибки маппинга;
•	подходы: снизу вверх, сверху вниз, большой взрыв, сэндвич-подход, постепенная интеграция;
•	для изоляции внешних сервисов используют тестовые окружения, контейнеры, mock-серверы и фикстуры;
•	интеграционные тесты обычно медленнее unit-тестов, поэтому их разделяют по наборам.
Пояснение
Актуальность интеграционного тестирования особенно высока в современных системах, где приложение состоит из множества сервисов и внешних API. Даже идеально протестированный модуль может дать сбой, если получает данные не в том формате или работает с другой версией контракта.
Вывод
Интеграционное тестирование подтверждает, что отдельные части продукта действительно образуют работающую систему.
 
Билет №17. Виды и актуальность разработки тестовой документации.
Краткий ответ
Тестовая документация — это набор документов и артефактов, которые описывают, что, как, зачем и с каким результатом проверяется в программном продукте. Она делает тестирование управляемым и воспроизводимым.
Основные положения
•	тест-план описывает цели, объем, ресурсы, сроки, риски, окружение и стратегию;
•	тест-кейс содержит шаги, входные данные, ожидаемый результат и условия выполнения;
•	чек-лист — краткий список проверок без детальных шагов;
•	баг-репорт фиксирует дефект: шаги воспроизведения, фактический и ожидаемый результат, окружение;
•	матрица трассировки связывает требования с тестами и показывает покрытие;
•	отчет о тестировании показывает результаты, найденные дефекты и оставшиеся риски.
Пояснение
Актуальность тестовой документации возрастает в командной работе, учебных проектах, коммерческой разработке и сопровождении продукта. Документация снижает зависимость от памяти конкретного тестировщика и помогает новым участникам быстрее понять, как проверяется система.
Вывод
Тестовая документация нужна для планирования, повторяемости, контроля качества и передачи знаний внутри команды.
 
Билет №18. Исключительные ситуации в работе программы. Основные виды и причины возникновения.
Краткий ответ
Исключительная ситуация — это событие, которое нарушает нормальный ход выполнения программы и требует специальной обработки. Исключение не всегда означает ошибку программиста: иногда это ожидаемая проблема внешней среды, например отсутствие файла или недоступность сети.
Основные положения
•	ошибки ввода: неверный формат, пустое обязательное поле, слишком большое значение;
•	ошибки ресурсов: файл не найден, нет прав доступа, закончилось место на диске;
•	ошибки сети: таймаут, потеря соединения, неверный ответ сервера;
•	ошибки выполнения: деление на ноль, обращение к null, выход за границы массива;
•	логические ошибки: данные допустимы технически, но нарушают бизнес-правила;
•	системные ошибки: нехватка памяти, сбой окружения, несовместимость библиотек.
Пояснение
Причины исключений могут быть внутренними и внешними. Внутренние связаны с ошибками кода или неправильной архитектурой. Внешние возникают из-за пользователя, файловой системы, сети, стороннего сервиса или оборудования. Обработка исключений должна сохранять устойчивость программы и давать понятную диагностику.
Вывод
Исключительные ситуации позволяют отделить обычную логику программы от обработки нештатных условий.
 
Билет №19. Назначение и варианты применения исключений, генерируемых пользователем.
Краткий ответ
Исключения, генерируемые пользователем, обычно понимаются как пользовательские, или собственные, исключения разработчика. Это специальные классы исключений, которые создаются для описания ошибок предметной области или конкретного приложения.
Основные положения
•	пользовательское исключение явно называет проблему: например, UserAlreadyExistsException;
•	по типу исключения видно, какое бизнес-правило нарушено;
•	собственные исключения удобно перехватывать отдельно от системных ошибок;
•	их применяют при проверке бизнес-логики, валидации данных, контроле состояния объекта;
•	исключение может хранить код ошибки, идентификатор операции, поле с ошибкой, текст для пользователя;
•	не стоит создавать отдельный класс исключения для каждой мелочи, если достаточно стандартного типа.
Пояснение
Хорошая практика — использовать исключения для действительно ошибочных ситуаций, а не для обычного управления потоком программы. Например, отсутствие пользователя в поиске может возвращаться как пустой результат, а попытка оплатить уже отмененный заказ может быть бизнес-исключением.
Вывод
Пользовательские исключения делают обработку ошибок более точной, выразительной и связанной с правилами конкретного приложения.
 
Билет №20. Функционал ИСР по обработке исключительных ситуаций.
Краткий ответ
Современные ИСР имеют встроенные инструменты для обнаружения, анализа и обработки исключительных ситуаций. Они помогают быстрее понять, где возникло исключение, какие данные к нему привели и как оно распространялось по стеку вызовов.
Основные положения
•	отладчик может остановить выполнение программы в момент возникновения исключения;
•	окно исключения показывает тип, сообщение, стек вызовов и свойства объекта исключения;
•	стек вызовов показывает цепочку методов, через которые программа пришла к ошибке;
•	просмотр переменных помогает понять состояние программы в момент сбоя;
•	exception breakpoints останавливают программу при возникновении определенного типа исключений;
•	ИСР подсвечивает потенциальные null-ошибки, необработанные исключения, неверные приведения типов;
•	быстрые исправления помогают добавить try-catch, пробросить исключение выше или создать обработчик.
Пояснение
Кроме отладки, ИСР помогает профилактически: статический анализ предупреждает о конструкциях, которые могут привести к исключениям. Интеграция с логами и тестами позволяет быстро перейти от упавшего теста или записи в журнале к проблемному участку кода.
Вывод
Функционал ИСР по работе с исключениями сокращает путь от симптома ошибки к ее причине.
 
Билет №21. Основные виды ошибок в программных продуктах.
Краткий ответ
Ошибки в программных продуктах можно классифицировать по этапу возникновения, проявлению и последствиям. Такая классификация помогает выбирать правильные способы поиска, тестирования и исправления.
Основные положения
•	синтаксические ошибки — нарушение правил языка программирования;
•	ошибки компиляции и сборки — неверные типы, отсутствующие зависимости, несовместимые версии библиотек;
•	ошибки времени выполнения — null reference, выход за границы массива, деление на ноль;
•	логические ошибки — программа выполняется, но результат неправильный;
•	интеграционные ошибки — неправильное взаимодействие модулей, API, базы данных или внешних сервисов;
•	ошибки требований — программа реализована, но не решает реальную задачу пользователя;
•	ошибки безопасности — уязвимости, утечки данных, неправильная авторизация;
•	ошибки производительности — медленная работа, утечки памяти, избыточные запросы.
Пояснение
Не все ошибки одинаково критичны. Для оценки используют серьезность и приоритет. Серьезность показывает влияние дефекта на систему, а приоритет — насколько срочно его нужно исправлять с точки зрения проекта.
Вывод
Знание видов ошибок помогает правильно организовать разработку, тестирование и сопровождение продукта.
 
Билет №22. Инспекция кода. Назначение и особенности применения.
Краткий ответ
Инспекция кода — это систематическая проверка исходного кода с целью обнаружения дефектов, нарушений стандартов, архитектурных проблем и потенциальных рисков до запуска программы или до попадания изменений в основную ветку.
Основные положения
•	инспекция может выполняться вручную: code review, парное программирование, формальные разборы;
•	автоматизированная инспекция включает статический анализ, линтеры, проверки форматирования и безопасности;
•	цель — найти ошибки раньше, чем они попадут в тестирование или эксплуатацию;
•	проверяют читаемость, стиль, корректность алгоритма, обработку ошибок, безопасность и тестируемость;
•	инспекция распространяет знания внутри команды и помогает договориться о едином стиле;
•	обсуждать нужно код и решение, а не личность автора.
Пояснение
Инспекция не заменяет тестирование. Тесты проверяют поведение программы, а инспекция оценивает сам код и проектные решения. Вместе они дают более высокий уровень качества.
Вывод
Инспекция кода — один из самых эффективных способов раннего обнаружения дефектов и повышения качества разработки.
 
Билет №23. Понятие качества программного обеспечения.
Краткий ответ
Качество программного обеспечения — это степень, в которой программа удовлетворяет требованиям, ожиданиям пользователей и условиям эксплуатации. Качественное ПО не только выполняет функции, но и делает это надежно, удобно, безопасно и сопровождаемо.
Основные положения
•	функциональная пригодность: система делает то, что требуется;
•	надежность: программа устойчиво работает при нормальных и частично ошибочных условиях;
•	производительность: система отвечает достаточно быстро и рационально использует ресурсы;
•	удобство использования: пользователь может выполнить свои задачи без лишних сложностей;
•	безопасность: данные защищены, доступ разграничен, уязвимости минимизированы;
•	совместимость: программа работает с нужными платформами, браузерами, устройствами и системами;
•	сопровождаемость: код можно изменять, тестировать, расширять и исправлять;
•	переносимость: продукт можно развернуть в разных средах.
Пояснение
Качество не возникает только на этапе тестирования. Оно закладывается в требованиях, архитектуре, стиле кода, тестах, документации, процессах ревью и эксплуатации. Исправлять дефекты после выпуска обычно дороже, чем предупреждать их во время разработки.
Вывод
Качество ПО — это комплексная характеристика продукта, процесса разработки и пригодности системы для реального использования.
 
Билет №24. Инструменты анализа качества программного продукта в среде разработки.
Краткий ответ
Среды разработки включают инструменты, которые помогают находить проблемы качества прямо во время написания кода. Часть таких инструментов встроена в ИСР, часть подключается как плагины или внешние анализаторы.
Основные положения
•	синтаксический и семантический анализ подсвечивает ошибки типов, переменных и недоступного кода;
•	инспекции кода ищут потенциальные null-ошибки, дублирование, избыточную сложность;
•	линтеры и форматтеры: ESLint, Pylint, Checkstyle, StyleCop, Prettier, Black;
•	статические анализаторы: SonarLint, Roslyn Analyzers, SpotBugs, PMD;
•	средства тестирования запускают unit- и интеграционные тесты, показывают упавшие проверки;
•	покрытие кода тестами показывает, какие части кода выполнялись тестами;
•	профилировщики анализируют время выполнения, память, запросы и узкие места;
•	рефакторинг помогает безопасно переименовывать, извлекать методы и менять структуру кода.
Пояснение
Важно не просто включить анализаторы, а настроить их под проект: выбрать правила, определить критичность предупреждений, включить проверки в CI. Иначе команда быстро привыкает игнорировать сотни некритичных сообщений.
Вывод
Инструменты анализа качества в ИСР помогают обнаруживать дефекты раньше и поддерживать единый технический уровень проекта.
 
Билет №25. Docker: назначение и основные возможности.
Краткий ответ
Docker — это платформа для контейнеризации приложений. Контейнер содержит приложение и его зависимости в изолированной среде, благодаря чему программа может одинаково запускаться на разных компьютерах и серверах.
Основные положения
•	Docker image — образ, то есть шаблон с приложением, библиотеками, настройками и инструкциями запуска;
•	Docker container — запущенный экземпляр образа;
•	Dockerfile — файл с инструкциями для сборки образа;
•	registry, например Docker Hub, хранит образы и позволяет обмениваться ими;
•	Docker Compose описывает и запускает несколько связанных контейнеров;
•	контейнеры удобны для одинаковой среды разработки, тестирования и развертывания;
•	Docker уменьшает проблему «у меня работает, а на сервере нет».
Пояснение
Docker не является виртуальной машиной в классическом смысле: контейнеры обычно легче и запускаются быстрее, потому что не несут полноценную гостевую ОС. При этом нужно учитывать безопасность образов, хранение секретов, размер образа, обновление зависимостей и сетевые настройки.
Вывод
Docker нужен для воспроизводимого запуска приложений и удобного управления зависимостями через контейнеры.
 
Билет №26. GitHub Actions: автоматизация рабочих процессов.
Краткий ответ
GitHub Actions — это встроенный в GitHub сервис автоматизации рабочих процессов. Он позволяет запускать сборку, тесты, проверки качества, публикацию артефактов и развертывание приложения по событиям в репозитории.
Основные положения
•	workflow — автоматизированный процесс, описанный YAML-файлом в .github/workflows;
•	event — событие, которое запускает workflow: push, pull_request, release, schedule, workflow_dispatch;
•	job — набор шагов, выполняемых на выбранном runner;
•	step — отдельное действие внутри job: команда, установка зависимостей, запуск тестов, готовый action;
•	runner — среда выполнения workflow, размещенная GitHub или настроенная самостоятельно;
•	GitHub Actions применяют для CI/CD: проверки кода, сборки, тестирования, публикации пакетов и деплоя;
•	секреты и переменные позволяют хранить токены и настройки, но требуют осторожного обращения.
Пояснение
Типовой сценарий: при каждом pull request GitHub Actions устанавливает зависимости, запускает линтер, выполняет тесты и сообщает результат в интерфейсе GitHub. Это не дает сломанному коду незаметно попасть в основную ветку.
Вывод
GitHub Actions автоматизирует повторяемые операции разработки и повышает надежность командного процесса.
 
Билет №27. API: понятие, типы и применение.
Краткий ответ
API, или Application Programming Interface, — это программный интерфейс, через который одна программа, модуль или сервис предоставляет свои возможности другим программам. API определяет, какие операции доступны, какие данные нужно передать и какой ответ можно получить.
Основные положения
•	API скрывает внутреннюю реализацию: клиенту достаточно знать правила обращения;
•	библиотечный API — набор классов, функций и методов библиотеки или фреймворка;
•	операционный API дает доступ к возможностям операционной системы;
•	Web API обеспечивает взаимодействие через сеть, чаще всего по HTTP/HTTPS;
•	REST API использует ресурсы и HTTP-методы: GET, POST, PUT, PATCH, DELETE;
•	SOAP API использует формализованные XML-сообщения и строгие контракты;
•	GraphQL API позволяет клиенту запрашивать именно нужные поля данных;
•	webhook — обратный вызов, когда одна система отправляет запрос другой при наступлении события.
Пояснение
API применяется для интеграции систем, мобильных приложений, веб-сервисов, микросервисной архитектуры, автоматизации бизнес-процессов и подключения внешних сервисов. Хороший API должен иметь документацию, понятные ошибки, версионирование, авторизацию, ограничения доступа и стабильные контракты.
Вывод
API — основной способ организовать управляемое взаимодействие между программами и информационными системами.
